# 02. 데이터 설계

## 2.0 데이터 기반 대화 (RAG 방식)

대화는 **데이터를 검색·조합한 뒤, 그 컨텍스트만 보고** LLM이 말하게 한다.

- **Retrieve**: 화자 캐릭터 정보, 최근 던전 로그, 메모리 요약, 직전 턴, 현재 이벤트/상황을 꺼낸다.
- **Augment**: 위 데이터를 프롬프트용 텍스트(슬롯)로 요약·조합한다.
- **Generate**: “이 컨텍스트 안에서만” 한 마디를 생성하도록 지시한다.

즉, **자유 연상이 아니라 “주어진 데이터를 기반으로만”** 대화한다. 데이터에 없는 주제는 억지로 꺼내지 않는다.

### 구현 원칙: 하드 코딩 금지

- **키워드·N값·상하한** 등은 코드에 문자열/숫자 리터럴로 넣지 않는다. 모두 **설정 데이터**(ScriptableObject, 설정 테이블, JSON 등)에서 로드한다.
- 아래 문서의 키워드 표·N값(8, 5, 3 등)·오프셋 ±값·상하한(-10~+10 등)은 **예시**일 뿐. 실제 값은 설정 에셋/테이블로 두고, 코드는 “설정을 읽어서 사용”만 한다.

## 2.1 캐릭터 기본 정보

| 항목 | 설명 | 예시 |
|------|------|------|
| 이름 | 길드원 식별자 | 카일, 리나, 브람 |
| 역할 | 전투/지원 역할 (말투·행동 힌트) | Vanguard, Support, Scavenger |
| 배경 | 짧은 인물 소개 (과거, 성향, 말투 근거) | "전직 기사. 선봉을 당연히 여김…" |

### 스탯·인벤토리 (Unity 연동용)

대화 맥락에서 "체력 낮다", "포션 몇 개 있다" 등을 쓰려면 **캐릭터별 현재 스탯·소지품**을 데이터로 넣는다.

| 구분 | 항목 | 설명 | Unity 매핑 |
|------|------|------|-------------|
| **스탯** | HP, MP, Atk, Def, Str, Dex, Int 등 | 현재/최대 HP·MP, 기본 스탯 | CharacterData (currentHP, maxHP, atk, def …) |
| **인벤토리** | 아이템 이름, 타입, 개수 | 소지품 요약(포션 N개, 유물 1개 등) | InventoryData.items + ItemData (itemName, itemType), 소모품은 Count |

- **용도**: RAG 컨텍스트의 [캐릭터] 또는 [상태] 블록에 "현재 HP/MP, 소지품: …" 형태로 넣어, LLM이 이 데이터만 보고 말하게 한다.
- **Unity에서**: 런타임에 CharacterData + InventoryManager(또는 InventoryData)를 읽어 위 구조로 채운 뒤 대화 요청 시 화자 캐릭터에 붙여 보낸다.

## 2.2 성격 수치 (10가지)

던전 행동 + 대화 톤/선택에 공통으로 사용.

| 수치 | 의미 (행동) | 대화 반영 |
|------|-------------|-----------|
| Courage | 위험 감수, 선진입 | 결단/공격적 표현 |
| Caution | 함정·미확인 회피, 포션 아낌 | 신중·안정 표현 |
| Greed | 루팅 우선, 무게 한계까지 | 이득/손해 언급 |
| Orderliness | 인벤토리 정리, 정돈 | 순서·기준 강조 |
| Impulsiveness | 계획 변경, 타겟 전환 | 즉흥·되물음 |
| Cooperation | 파티 유지, 지원 스킬 | 같이/협력 표현 |
| Aggression | 스킬·평타 적극 사용 | 단호한 말투 |
| Focus | 목표 유지, 한눈 안 팜 | 주제 고수 |
| Adaptability | 패턴 학습, 전술 전환 | 입장 전환 가능성 |
| Frugality | 소모품 아낌, 비상용 보존 | 손익·효율 언급 |

- **범위**: 0~100 (또는 -100~100). 기본값 50 전후.
- **용도**: 톤 선택(careful/bold/polite 등), 말투 가이드, 주제 이탈 억제에 활용.

### 성격 수치 → 말투 가이드 (프롬프트용 요약)

수치를 그대로 넣지 않고, 상위/하위 몇 개를 골라 1~2문장으로 요약해 [캐릭터] 말투 가이드에 넣는다.

| 수치가 높을 때 | 말투 키워드/문장 예 |
|----------------|----------------------|
| Courage, Aggression | bold, 단호한 말투, 결단적 |
| Caution, Frugality | careful, 신중·효율 언급 |
| Greed | 이득/손해·루팅 언급 |
| Orderliness, Focus | 순서·기준·목표 강조 |
| Impulsiveness | 즉흥·되물음·전환 빠름 |
| Cooperation | 같이/협력·제안형 |
| Adaptability | 입장 전환·타협 표현 |

- 구현: 상위 2~3개 수치만 골라 "말투: ~하고 ~한 편이다" 형태로 문자열화. (토큰 절약)

## 2.3 던전 로그 (기억용)

한 번의 던전 플레이 결과를 **구체적으로** 적은 로그. 대화에서 "아까 2층 함정", "유물 나온 거", "리나가 포션 썼지"처럼 끌어쓸 수 있어야 하므로 이벤트는 한 줄씩 상세히.

| 항목 | 설명 |
|------|------|
| 던전명 | 예: 잊힌 묘지, 습지 동굴 |
| 층/구간 | 예: 2층, 보스방 |
| 이벤트 목록 | **구조화된 데이터만** (줄글/Summary/Note 금지). 한 건당 객체에 EventType + 타입별 필드만 사용 |
| 시각(선택) | 언제 플레이했는지 |
| 결과(선택) | 클리어/퇴장/보스 격파 등 |
| 파티원(선택) | 함께 플레이한 멤버 ID 또는 이름 |

- **저장**: 길드원별로 최근 N회분만 유지. (N은 **대화 설정**에서 로드. 예: 5, 축소 시 3)
- **대화 활용**: 이벤트 객체를 프롬프트용 한 줄로 **포맷만** 해서 RAG에 넣음. 저장은 항상 구조화 데이터만.

**이벤트 타입별 필드 (줄글 없음)**

| EventType | 필드 | 예시 데이터 |
|-----------|------|-------------|
| Trap | Location, TargetId, Damage, HpBefore, HpAfter | Location: "동쪽 복도", TargetId: "카일", Damage: 15, HpBefore: 100, HpAfter: 85 |
| Combat | **Enemies**: `[{ Name, Count }]`, ActorId, Turns, BlockCount? | Enemies: [{ "Name": "스켈레톤", "Count": 2 }], ActorId: "카일", Turns: 2 |
| Artifact | ItemName, Location | ItemName: "잊혀진 룬", Location: "보스방 직전 방" |
| Heal | ActorId, TargetId, HpBefore, HpAfter | ActorId: "리나", TargetId: "카일", HpBefore: 85, HpAfter: 100 |
| ConsumePotion | ActorId, ItemName, ItemCount, MpBefore, MpAfter | ActorId: "리나", ItemName: "회복 포션", ItemCount: 1, MpBefore: 20, MpAfter: 45 |
| Loot | **LootItems**: `[{ ItemName, Count }]`, Location?, ActorId? | LootItems: [{ "ItemName": "골드", "Count": 80 }, { "ItemName": "부러진 단검", "Count": 1 }] |
| TrapAvoided | Location?, AvoidedCount? | Location: "동굴 입구", AvoidedCount: 1 |
| ItemTransfer | FromId, ToId, ItemName, ItemCount | FromId: "브람", ToId: "카일", ItemName: "회복 포션", ItemCount: 2 |
| InventorySort | ActorId, Location? | ActorId: "브람", Location: "보스방 앞 방" |
| PartyCheck | ActorId, CheckType | ActorId: "리나", CheckType: "HpMp" 또는 "PotionCount" |
| DebuffClear | ActorId, TargetId, ClearCount | ActorId: "리나", TargetId: "카일", ClearCount: 1 |

모든 이벤트에는 공통 메타정보로 **Order**(이벤트 순서, 0/1/2… 또는 1/2/3…)와  
**TimeOffsetSeconds**(던전 입장 시점으로부터의 상대 시간, 초 단위)를 둘 수 있다.  
배열 인덱스로도 순서를 알 수 있지만, 외부 로그/리플레이와 동기화하거나 필터링할 때 Order/TimeOffsetSeconds를 사용한다.

## 2.4 대화 메모리 (세션·장기)

| 구분 | 내용 | 용도 |
|------|------|------|
| 세션 턴 | 현재 대화의 최근 발화 목록 (화자 + 텍스트) | 직전 맥락, 반복 방지 |
| 장기 요약 | 관계/신뢰/기분 오프셋, 미해결 안건 등 짧은 요약 | 다음 대화 시 힌트 |
| 시나리오 키 | RecentEvent + CurrentSituation 기반 키 | 장면 바뀌면 턴/안건 정리 |

- **시나리오 전환**: 이벤트/상황이 크게 바뀌면 세션 턴·일부 메모리 초기화해 주제 꼬임 방지.

### 장기 메모리 갱신 규칙

- **시점**: 턴 한 번 생성·저장한 직후 (또는 N턴마다 배치).
- **입력**: 해당 턴의 `intent`·`line`(및 선택적으로 tone)을 보고 갱신.
- **갱신 내용**: 관계/신뢰/기분 오프셋, 미해결 안건. 1~2문장 요약으로 유지.
- **담당**: 대화 매니저(또는 별도 메모리 모듈)가 턴 저장 시 호출.

**관계/신뢰/기분 오프셋 계산 (키워드 기반)**  
`line`·`intent`에 아래 키워드가 포함되면 해당 오프셋을 소폭 적용(예: ±1~2). 누적 후 상·하한 있음(예: -10~+10).

| 방향 | 예시 키워드(긍정) | 예시 키워드(부정) |
|------|-------------------|-------------------|
| **관계**(GM↔길드원) | 협력, 지시 따름, 알겠어, 도와줘서 고마워 | 갈등, 불만, 왜 그래, 말 안 들어 |
| 신뢰 | 감사, 도움, 함께 잘했다, 믿어 | 무시, 거절, 혼자만, 의심 |
| 기분 | 좋아, 잘됐다, 다행 | 화남, 실패, 아쉬워, 위험했어 |

- **구현(하드 코딩 금지)**: 위 표는 **예시**. 실제 키워드 목록·오프셋 ±값·상하한은 반드시 **설정 데이터**(SO, 설정 테이블, JSON)에서 로드. 코드에는 `new[] { "감사", "도움" }` 같은 리터럴을 넣지 않는다. 매 턴 설정에서 읽은 목록으로 `line` 매칭 후 오프셋 누적.

## 2.5 대화 요청 시 필요한 입력

한 턴을 생성할 때 필요한 정보:

- **화자**: 어떤 길드원이 말하는가
- **상대**: 길드 마스터인가, 다른 길드원인가
- **직전 발화**: 상대의 마지막 한 마디 (또는 길드 마스터의 질문/지시)
- **최근 이벤트**: "오늘 던전에서 ~ 있었음" 수준의 요약
- **현재 상황**: "아지트 식당에서 식사 중" 등
- **해당 캐릭터**: 기본 정보 + 성격 수치 + 배경 + 최근 던전 로그 요약 + 메모리 요약

### RAG 컨텍스트로 쓸 데이터 목록

| 데이터 | 출처 | 프롬프트에 넣는 형태 |
|--------|------|----------------------|
| 캐릭터 | 2.1, 2.2 | 이름, 역할, 배경, 성격 수치·말투 가이드 |
| 던전 로그 | 2.3 | 최근 N회 요약 문자열 (예: "[잊힌 묘지 2층] 함정 1회, 유물 획득…") |
| 메모리 | 2.4 | 장기 요약 + 직전 턴 리스트 (최근 N턴, **대화 설정**에서 로드. 예: 8, 축소 시 5) |
| 현재 맥락 | 입력 | 최근 이벤트, 현재 상황, 직전 발화(상대 한 마디) |

- **N 값**: 직전 턴 수, 던전 로그 회수 등은 **대화 설정(Config)** 에서 읽는다. 코드에 8, 5, 3 등 매직 넘버를 넣지 않는다. [04-LLM-연동-설계](04-LLM-연동-설계.md)의 토큰 우선순위와 함께 조정.

→ **이 네 가지를 합친 블록이 곧 “검색된 문서”**이고, LLM은 여기 있는 내용만 참고해 답한다.

## 2.6 대화 설정(Config) 스키마

구현 시 **하드 코딩 금지**를 지키기 위해, 아래 필드를 가진 설정을 한 곳(JSON, SO, 테이블)에서 로드한다.

| 필드(예시) | 용도 | 예시 값 |
|------------|------|---------|
| `MaxRecentTurns` | 직전 턴 수 N | 8 |
| `MaxDungeonLogs` | 던전 로그 최대 회수 | 5 |
| `MaxRecentTurnsFallback` | 토큰 부족 시 턴 수 | 5 |
| `MaxDungeonLogsFallback` | 토큰 부족 시 로그 회수 | 3 |
| `MaxRetryCount` | 리트라이 최대 횟수 | 2 |
| `OffsetMin`, `OffsetMax` | 오프셋 누적 상·하한 | -10, 10 |
| `OffsetStep` | 키워드 1회 매칭 시 ± | 1 또는 2 |
| `KeywordCategories` | 관계/신뢰/기분별 키워드 목록(긍정/부정) | [02 2.4 표 참고] |
| `AllowedTopicKeywords` | 주제 이탈 검사용 허용 주제 목록 | 유물, 포션, 함정, … |
| `FallbackToneByRole` 또는 캐릭터별 `FallbackLines` | 폴백 시 사용할 말투/문장 목록 | 역할별 또는 캐릭터 ID별 문자열 목록 |
| (선택) `KeywordExtractMinLength` | 핵심어 최소 글자 수 | 2 |

- 캐릭터 데이터에 `FallbackLines`를 두거나, 설정에 역할별 기본 문장 목록을 두어 폴백 시 그중 하나를 선택해 사용.
