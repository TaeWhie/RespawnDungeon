# 08. 런타임 시퀀스 상세

구현 기준: `DialogueManager.cs`, `Program.cs`. **이 문서는 03보다 한 단계 낮은 추상화**로, 실제 호출 순서와 하드코딩을 적는다.

---

## 8.1 `InitializeAsync` (메뉴 1·2 공통)

순서는 대략 다음과 같다.

1. `LoadCharactersWithJobSkillFilter()` — `JobSkillRules`로 스킬 정리.
2. `_charMap` — `Name`과 `Id`(있으면) 모두 키로 등록.
3. `LoadWorldLore`, `LoadItemDatabase`, `GameReferenceBundle` 구성(몬스터·함정·스킬·이벤트·파티·직업 등).
4. `LoadTimelineData()` → `ActionLog`를 `Order`로 정렬.
5. `ActionLogBuilder.Build` → 던전 런별 `DungeonLog`, `BaseLogEntry` 분해.
6. `MemoryManager.BuildEpisodicBuffer` — 에피소드 텍스트(던전 서사) 구축.
7. `RebuildPerspectiveMemoryCache` — `ActionLog` + 캐릭터로 화자별 관점 줄(설정 켜진 경우).
8. `OllamaEmbeddingClient` 생성, `EmbeddingKnowledgeIndex.BuildAsync`  
   - 입력: 월드 로어, 몬스터, 함정, 스킬, 아이템, 캐릭터(전처리 청크).  
   - 실패 시 **예외**로 중단(메시지에 `EmbeddingModel` 안내).
9. `UseGuildOfficeSemanticGuardrail`이면 `GuildOfficeSemanticGuardrail.WarmupAsync` — 실패 시 콘솔에 “생략”만 하고 **null**(집무실은 휴리스틱·키워드 폴백 경로로 진행 가능).

## 8.2 메뉴 1 — 아지트 허브 (`BaseHubScreen` → `RunBasePairDialogueAsync`)

### 8.2.1 전제

- `master` 제외 후 **아지트에 있는 동료**가 2명 이상 — `BaseHubScreen.FilterAtBase`: `CurrentLocationId`가 비어 있거나 `BaseDatabase.json`의 `BaseId`와 일치.
- 인원 부족 시 콘솔에 오류만 출력하고 종료.

### 8.2.2 활동 배정

- `BuildActivityPool`: 각 시설 `AvailableServices`를 쉼표로 나눠 `"식당 — 식사"` 형태로 넣고, 시설별 `"… — 시설 점검"` 및 공통 잡무 문자열을 풀에 합친다.
- 캐릭터 목록을 섞은 뒤 **앞의 2명**을 대화 쌍으로 고르고, 나머지에게는 풀에서 무작위 활동 문자열을 부여한다.
- 대화 쌍 둘 다 활동 표시는 `"[시설명]에서 동료와 대화 중 (↔ 상대이름)"` 형태로 **같은 이벤트가 정확히 한 쌍**에만 매칭된다.

### 8.2.3 대화 실행

- `utteranceCount = Random(2..4)` — 두 사람이 **번갈아** 말하는 **총 줄 수**(2~4). 첫 화자는 무작위. 홀수면 한 사람이 한 줄 더 말함.
- `RunBasePairDialogueAsync`: `SynthesizeSocialImpressionsAsync` 후 턴마다 `PromptBuilder` + RAG + `PartyRosterResolver` — 구 6인 관전과 동일한 **2인 전용** 루프.
- 종료 시 `PersistDialogueSessionToActionLogIfEnabled` 라벨 `"아지트 동료 대화(2인)"`, `UpdateSocialSettlementAsync` 호출.

## 8.3 메뉴 2 — 길드장 집무실 (`RunGuildMasterSessionAsync`)

### 8.3.1 전제

- `master` 제외한 캐릭터가 1명 이상.

### 8.3.2 동료 선택

- `TryPromptGuildOfficeBuddySelection` — 번호, 이름, Id로 선택. `exit` 등은 집무실 종료.
- **동료를 바꿀 때마다** `ClearWorkingMemory()` — 맥락 섞임 방지.

### 8.3.3 입력 루프

- `ReadConsoleLine` — Windows에서는 `ReadConsoleW` 우선(버퍼 길이·널 종료 처리). 비Windows/실패 시 `Console.ReadLine`.
- 입력 정규화: BOM·제로폭 제거(`NormalizeConsoleInputLine`).
- `exit`류(`GuildOfficeTopicGate.IsGuildOfficeExitRoomCommand`) → 집무실 나감.
- `change` / `바꿔` 등 → **내 루프만** 빠져나가 상대 선택으로.
- 그 외: 워킹 메모리에 길드장 발화 추가, `GenerateGuildOfficeBuddyReplyAsync` 호출.

### 8.3.4 `GenerateGuildOfficeBuddyReplyAsync` 핵심

1. `GuildOfficeTopicGate.ResolveGuildOfficeSignalsAsync` — 변칙 종류 + 메타/오프/원정 코사인.
2. `ComputeDeepExpeditionForGuildOffice` — “두꺼운 원정 프롬프트” 켤지(임베딩·키워드·설정 다수).
3. `deepExpeditionEffective = deepExpedition && !frustrationStop`(짜증/중단 룩어_like면 원정 블록 억제).
4. **일상 턴**(`!deepExpeditionEffective`):  
   - `episodicForPrompt`는 고정 안내 문구로 **던전 서사 생략**.  
   - `latestLogForPrompt`도 생략 안내.  
   - `perspectiveForPrompt` null, **`ragBlock` 빈 문자열**(RAG 호출 자체를 안 함).
5. **원정 턴**: 관전 모드와 유사하게 Episodic·ActionLog·관점·RAG 전부 활성.
6. `AethelgardMcpRuntimeFacts.Build` — `inventory://` 등.
7. `PromptBuilder.BuildSystemPrompt` — 길드장 1:1 전용 인자 다수(`guildMasterDeepExpeditionTurn` 등).
8. `BuildGuildMasterInteractiveUserPrompt`.
9. Ollama → JSON `line` 필수. 실패 시 `GuildOfficeLlmTurnResult`에 에러 메시지.
10. 성공 시 `CharacterMoodUpdater.ApplyAfterGuildOfficeTurn`, 선택적으로 워킹 메모리·세션 로그.

### 8.3.5 종료 후

- `PersistDialogueSessionToActionLogIfEnabled` — 라벨 `"길드장 상호작용"`.
- `UpdateSocialSettlementAsync` — 길드장 세션은 **전체 턴 텍스트**를 `FormatDialogueTurnsForSettlement`로 넘김.

---

## 8.4 격리 턴 (`RunIsolatedGuildOfficeLlmTurnAsync`)

탐색/테스트용: 워킹 메모리 클리어, 상대 마지막 대사 제거, 길드장 유저 한 줄만 넣고 `GenerateGuildOfficeBuddyReplyAsync(..., persistBuddyLineToSession: false)` — **ActionLog 세션 누적 없음**.

---

## 8.5 부가: `InferBaseLocationFromWorldState`

대화 세션을 ActionLog에 넣을 때 `worldState` 문자열에 "집무실"·"식당" 등이 포함되면 `Location` 후보를 고른다.
